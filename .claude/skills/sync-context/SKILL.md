# Sync Context (Pika)

用于**一次大功能开发与测试结束后**做统一收口：把“当前状态、验证结论、下一步优先级”同步进 memory 与项目文档，确保重启电脑或新开 Claude 对话时可以快速接着干活。

## 目标
1. **可续聊**：新会话能在 1-2 分钟内恢复上下文。
2. **有结论**：记录本轮“做了什么、验证到哪、还差什么”。
3. **不臃肿**：每次同步都做精简去重，避免文档越写越长。

## 必同步位置
- **Auto-memory**：`C:\Users\saiy\.claude\projects\d--Documents-Projects-Pika\memory\`
  - `MEMORY.md`（索引，一行一条）
  - 会话收口文件：`session-YYYY-MM-DD-*.md`
  - 相关长期记忆（project/feedback/reference）按需更新
- **项目文档**（仓库）
  - `docs/PROJECT_STATUS.md`：**唯一 backlog 入口（P0/P1/P2）+ 当前状态**
  - `docs/IMPLEMENTATION_PLAN.md`：当前范围与验收口径（不重复维护 backlog）
  - `CLAUDE.md`：仅在规则/架构约束发生变化时更新

## 标准流程
1. 先读当前 memory + docs，避免重复和口径冲突。
2. 提炼“本轮值得持久化”的信息：
   - 决策及原因
   - 部署/联调事实
   - 验证结论（通过/未通过）
   - 下一步优先级
3. 更新或新建会话收口 memory（必做）：
   - 只写：当前状态、已验证结论、下一步
   - 文件名清晰可检索（日期+主题）
   - 在 `MEMORY.md` 加索引
4. 回写 docs：
   - `PROJECT_STATUS.md` 维护为唯一 backlog（P0/P1/P2）
   - 完成项从 backlog 移除，未完成项按优先级重排
   - `IMPLEMENTATION_PLAN.md` 仅更新范围/验收口径，不重复 backlog
   - 删除过时方案与重复叙述
5. 输出简报（3-5 行）：
   - 改了哪些文件
   - 精简了什么
   - 当前下一步

## 精简规则（每次必做）
- 删除代码可直接推导的信息（接口清单、字段清单、文件清单）。
- 合并重复 memory，避免同一事实在多个文件重复描述。
- 用结论句替代过程叙述；同一信息只保留一处主记录。

## 安全规则
- 不写入任何真实密钥/令牌/内网敏感入口。
- 敏感配置只保留“在 NAS `.env` 注入”的事实描述。

## 注意
- 相对日期转绝对日期。
- 若用户说“某段不要记忆”，必须排除。
- 即便无大进展，也要执行一次精简检查并说明“本次无新增，仅清理”。

## 收口模板（新增）
每次会话收口 memory 建议按下面结构写：

1. **当前状态**：一句话说明本轮到达的可用状态。
2. **已验证结论**：2-5 条，仅写“通过/未通过 + 影响”。
3. **下一步优先级**：最多 3 条，按 P0/P1/P2。
4. **风险/阻塞**（可选）：仅写仍会卡住下轮推进的项。

> 原则：只写结论，不写过程流水账。

## 最终报告格式（新增）
执行完 `sync-context` 后，给用户固定简报：

1. **已更新**：列出更新的 memory/docs 文件名。
2. **已精简**：说明删了哪些重复/过时口径。
3. **下一步**：给出当前最优先 1-3 项。
4. **新窗口开场白**：提供可直接复制的一句话。

## 新窗口开场白模板（新增）
给用户一条可直接复用的开场白：

`请先读取 memory 与 docs，按最新会话收口继续：先用 3 句话告诉我当前状态、已验证结论、下一步优先级，然后直接从 P0 开始执行。`
