# Pika 就医服务实施计划（当前）

> Backlog 总览维护在 `PROJECT_STATUS.md`；单项 feature / bug 明细维护在 `docs/tickets/*.md`；本文件只保留范围、共享验收口径与跨 ticket 实施包。

## 当前范围（已完成）

1. `hospital` 全链路贯通（上传/落库/查询/展示）
2. 一份报告多图（`image_paths`）
3. 两阶段上传（草稿预解析 → 人工修正 → 提交）
4. 平台能力：服务市场、成员管理、个人资料、未登录拦截
5. 成员邀请：小程序内二维码扫码加入 + 邀请码兜底
6. 公网可用：Azure VM + frp + Caddy，体验版可触达后端
7. 就医浏览体验收口：趋势页单成员优先，历史页默认本人 + 近 3 个月 + 医院多选
8. 上传页与权限设置页 UI/UX 收口（大图/卡片/状态卡样式统一）

## 当前验收口径

1. 邀请闭环：管理员生成二维码 → 被邀请人从“我”页扫码 → 确认加入。
2. 登录稳定：连续登录/退出/重登首轮成功率达标。
3. 资料稳定：头像更换后即时回显与回页一致。
4. 权限边界：成员管理非管理员只读；就医服务仅同一家庭内默认互相开放既有全部权限，跨家庭仍不可见不可操作。
5. 上传稳态：识别与提交失败时自动重试 2 次，按钮状态可感知。
6. 导航闭环：上传成功后用户可一跳回到“就医首页/历史”。
7. 浏览验收：趋势页与历史页的筛选流符合“先成员、后数据”的单人优先逻辑。
8. 成员选择一致性：上传、历史、趋势、权限设置的成员选择 UI/UX 与提示词保持一致，并按场景联动就医权限。

## Ticket 协作约定

1. `PROJECT_STATUS.md` 只保留高层总览与优先级，不写单项实现细节。
2. 一个 feature / bug 如果会跨多个 loop，就先建 `docs/tickets/TKT-xxx-*.md`。
3. loop 结束后的进展、阻塞、下一步，优先更新对应 ticket 的“Loop 记录”。
4. 只有当 ticket 影响全局范围、跨 ticket 顺序或共享验收口径时，才回写本文件。
