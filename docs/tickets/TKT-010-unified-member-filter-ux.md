# TKT-010｜就医成员选择器 UI/UX 与权限联动统一

## 基本信息

- ID：`TKT-010`
- 类型：feature
- 优先级：P0
- 状态：done
- 负责人：待定
- 关联阶段：体验版稳定性回归
- 关联入口：`docs/PROJECT_STATUS.md` / P0-5

## 背景

- 用户反馈 / 现象：就医服务里的上传检查单、检查历史、指标趋势、权限设置都存在按家庭成员选择/筛选的能力，但 UI/UX 样式、提示词和默认行为不一致。
- 当前原因判断：各页面按自身流程分别实现成员 filter，缺少统一的成员选择组件、文案规范和权限联动规则。
- 为什么现在做：成员选择是就医服务高频入口，样式和规则不一致会放大理解成本，也容易导致用户不知道自己能查看或代上传谁的数据。

## 目标

1. 统一就医服务内成员选择/筛选控件的视觉样式、提示词和交互反馈。
2. 明确上传、历史、趋势、权限设置四个入口各自的默认成员与可选成员规则。
3. 选择他人时按场景联动权限判断，让不可用成员有一致、可解释的反馈。

## 非目标

1. 不重做家庭成员管理页。
2. 不在本 ticket 内修改底层 ACL 策略本身；默认授权修复归 `TKT-004`。
3. 不扩展就医服务以外模块的成员选择体验。
4. 不强制一次决定“卡片”或“下拉框”为最终方案；实现前可根据页面密度选定统一规范。

## 范围

### 要做
- 盘点上传检查单、检查历史、指标趋势、权限设置页现有成员选择 UI、默认值、空态和提示词。
- 设计并落地一套统一成员选择 UI 规范，可选方案包括卡片式选择或统一下拉框样式。
- 上传检查单：默认选择自己；可选择他人；选择他人时需要判断当前用户是否有 `upload_for_owner` 权限。
- 检查历史：默认筛选自己；选择他人时需要判断当前用户是否有 `view_report` 权限。
- 指标趋势：默认筛选自己；选择他人时需要判断当前用户是否有 `view_report` 权限。
- 权限设置：不提供“自己”作为可设置对象；默认可为空或默认展示第一个其他成员；按选择对象加载下方权限设置。
- 统一不可选、无权限、加载中、空家庭成员等状态的提示方式。

### 不做
- 新增家庭角色体系。
- 改造报告上传、历史查询、趋势查询的核心数据结构。
- 重新设计就医首页或全局导航。
- 替代 `TKT-004` 的默认全权限策略修复。

## 方案备注

- 涉及模块：上传检查单页、检查历史页、指标趋势页、权限设置页、可能的成员选择公共组件/工具函数、就医 ACL 权限读取。
- 关键实现点：统一控件样式与提示词；默认成员逻辑一致；权限判断要和后端实际能力一致；权限设置页排除自己并支持空态。
- 风险 / 待确认：统一为卡片还是下拉框需要结合页面空间、移动端触达成本和现有样式决定；若依赖 `TKT-004`，需要明确先后顺序。

## 验收口径

1. 上传检查单、检查历史、指标趋势、权限设置四个页面的成员选择控件视觉风格和提示词一致。
2. 上传检查单默认选择自己；选择他人时，只有具备代上传权限才允许继续上传。
3. 检查历史和指标趋势默认选择自己；选择他人时，只有具备查看权限才展示对应数据。
4. 权限设置页不出现“自己”作为被设置对象；无默认对象时有明确空态，选择成员后再加载并展示该成员对我的权限。
5. 无权限、加载失败、无其他家庭成员等状态有一致且可理解的反馈。

## 部署影响

- 前端：需要，多个就医页面的成员选择 UI 与交互会改。
- 后端：可能需要，若现有接口不能支撑前端判断成员权限状态。
- 数据库：无，除非实现时发现需补充持久化权限字段；优先复用现有 ACL。
- 配置 / NAS：无。
