# TKT-013｜一次上传多类检查单只识别一张

## 基本信息

- ID：`TKT-013`
- 类型：bug
- 优先级：P0
- 状态：todo
- 负责人：待定
- 关联阶段：就医能力升级 / 上传识别可靠性
- 关联入口：`docs/PROJECT_STATUS.md` 对应条目
- 依赖：`TKT-007` 检查单类别识别与关注指标联动

## 背景

- 用户反馈 / 现象：用户在上传检查单时，一次选择了多张图片，但这些图片属于不同检查类别，例如血常规、肝功等；最终系统只识别出一张，其他类别的检查单没有被识别成独立结果。
- 当前原因判断：现有“一份报告多图”能力默认多张图片属于同一份报告，识别和提交链路缺少“多张图片可能代表多份不同类别检查单”的拆分、分类和多报告生成能力。
- 为什么现在做：该问题会造成一次上传中部分检查单被漏识别，属于用户感知很强的数据丢失/识别遗漏问题。

## 目标

1. 一次上传多张图片时，如果图片分别属于不同检查类别，系统能识别并拆分为多份报告或给出明确的人工拆分/确认入口。
2. 至少覆盖血常规、尿常规、肝肾功、肿瘤标志物、甲功等 `TKT-007` 定义的核心类别。
3. 避免只返回其中一张或一类结果，导致其他检查单静默丢失。

## 非目标

1. 不在本 ticket 内扩展无限多检查类别；类别基线依赖 `TKT-007`。
2. 不重做完整上传 UI 流程，除非多报告确认所必需。
3. 不保证首版能自动完美拆分所有混合上传场景，必须保留人工确认或修正兜底。

## 范围

### 要做
- 明确与 `TKT-007` 的依赖关系：先具备检查单类别识别基线，再处理多图拆分与多报告生成。
- 盘点当前上传草稿、识别、提交链路对“一次多图=一份报告”的假设。
- 支持一次上传中识别出多个检查类别时，生成多份可确认的报告草稿或明确提示用户拆分处理。
- 保证每张上传图片都有处理结果：归入某份报告、提示无法识别、或进入人工确认。
- 增加防静默丢失反馈，避免只展示第一张/第一类识别结果。

### 不做
- 全量医学文档分类体系。
- 与上传无关的历史/趋势展示重构。
- 跳过 `TKT-007` 直接硬编码零散类别规则。

## 方案备注

- 涉及模块：上传页、多图草稿识别、视觉识别后处理、报告草稿提交、报告类别字段/展示。
- 关键实现点：`TKT-007` 提供类别识别基线；本 ticket 处理混合类别多图的拆分策略、用户确认体验和多报告提交。
- 风险 / 待确认：如果识别模型返回的是整批图片的单一结构，需要调整 prompt / 后处理，使其能返回按图片或按类别分组的多报告结构。

## 验收口径

1. 一次上传血常规 + 肝功等不同类别图片时，不会只识别其中一张或一类。
2. 系统能展示多份报告草稿，或明确提示用户需要拆分/确认每张图片归属。
3. 每张上传图片都有可追踪状态，不出现静默丢失。
4. 与 `TKT-007` 的五类检查单类别规则一致，类别信息能被后续历史/趋势使用。
5. 单一类别多图仍可作为同一份报告处理，不因修复混合上传而回归。

## 部署影响

- 前端：需要，上传草稿确认页可能需要支持多报告草稿或拆分提示。
- 后端：需要，识别后处理、草稿结构和提交逻辑可能会改。
- 数据库：可能需要，取决于多报告草稿结构和类别字段方案；应与 `TKT-007` 的 schema 设计对齐。
- 配置 / NAS：无，除非新增识别 prompt 或模型配置项。
