# TKT-014｜局域网/公网后端地址自动切换

## 基本信息

- ID：`TKT-014`
- 类型：feature
- 优先级：P1
- 状态：todo
- 负责人：待定
- 关联阶段：网络连接体验 / 部署可用性
- 关联入口：`docs/PROJECT_STATUS.md` 对应条目

## 背景

- 用户反馈 / 现象：当前小程序后端地址默认使用公网 HTTPS 域名；开发环境可通过 `config.local.js` 手动覆盖到局域网地址，但手机或微信开发工具在家庭局域网内时，用户希望优先使用 `192.168.1.200`，手机在外使用蜂窝网络时自动使用 Pika 的公网域名。
- 当前原因判断：`miniprogram/config.js` 当前导出静态 `BASE_URL`，请求封装直接拼接该地址；缺少运行时网络环境判断、局域网连通性探测和公网兜底策略。
- 为什么现在做：在家中局域网访问 NAS 后端通常更快、更稳定，也能减少公网链路依赖；外网场景又必须自动回到公网域名，避免用户手动改配置。

## 目标

1. 小程序在家庭局域网内时，优先使用局域网后端地址，例如 `192.168.1.200`。
2. 小程序在蜂窝网络或非家庭网络时，自动使用 Pika 的公网域名。
3. 自动判断失败时有稳定兜底，不能因为探测局域网失败而阻塞登录、上传、历史、趋势等主流程。
4. 保留开发工具/开发版可配置能力，避免把真实公网标识或敏感配置写死到不合适的位置。

## 非目标

1. 不在本 ticket 内重做后端公网部署、frp、Caddy 或证书体系。
2. 不支持任意用户自定义多套家庭网络规则；首版只服务当前 Pika 家庭网络。
3. 不绕过微信小程序合法域名、HTTPS、局域网调试等平台限制。
4. 不在业务页面里散落网络判断逻辑，切换能力应集中在配置/请求层。

## 范围

### 要做
- 设计小程序后端 endpoint 选择策略，覆盖局域网地址、公网域名、探测超时和兜底。
- 利用已知家庭网络信息作为判断线索：家庭 IP 段 `192.168.1.*`，Wi-Fi 名称 `CMCC-Saiy` / `CMCC-Saiy-5G`。
- 评估微信小程序可用的网络判断能力，例如网络类型、Wi-Fi 信息权限、局域网 `/health` 探测、请求失败后公网 fallback。
- 将请求层从静态 `BASE_URL` 调整为可在运行时选择 endpoint 的机制。
- 增加缓存策略，避免每个请求都重复探测局域网；网络变化时能重新判断。
- 明确微信开发工具、开发版、体验版/正式版下局域网地址访问的限制和配置要求。
- 在局域网不可达、SSID 获取失败、权限被拒、蜂窝网络等场景下稳定使用公网域名。

### 不做
- 不把真实密钥或不应入仓的公网敏感配置写入代码。
- 不为所有服务模块分别实现网络切换。
- 不把后端 API 路由改成两套不同协议。

## 方案备注

- 涉及模块：`miniprogram/config.js`、`miniprogram/utils/request.js`、`miniprogram/utils/auth.js`、可能的 `/health` 探测与本地配置示例。
- 关键实现点：优先考虑“局域网健康检查 + 超时兜底公网”的稳健策略；SSID/IP 段可作为辅助信号，但受微信权限和平台限制，不能作为唯一依据。
- 风险 / 待确认：微信小程序正式/体验环境对 `http://192.168.1.200`、局域网 IP、HTTPS 证书和 request 合法域名有约束；如果局域网地址不能在真机环境直接请求，可能需要局域网 HTTPS 域名或仅在开发/调试态启用。

## 验收口径

1. 手机连接 `CMCC-Saiy` 或 `CMCC-Saiy-5G` 且 `192.168.1.200` 后端可达时，小程序请求走局域网地址。
2. 手机切换到蜂窝网络或非家庭 Wi-Fi 时，小程序请求自动走公网域名。
3. 局域网地址探测失败或超时时，请求能自动回落到公网域名，主流程不被卡死。
4. 微信开发工具和真机至少各验证一次网络切换行为，记录平台限制和必要配置。
5. 登录、上传、历史、趋势等依赖请求封装的主链路都使用同一套 endpoint 选择结果。

## 部署影响

- 前端：需要，小程序配置和请求封装会改。
- 后端：可能需要，若 `/health` 或局域网 HTTPS 支持不足，需要补健康检查或部署配置。
- 数据库：无。
- 配置 / NAS：可能需要，需确认局域网地址、端口、HTTPS/证书、微信合法域名或开发调试配置。
